## Project Overview

VeeCode Platform documentation site built with **Docusaurus 3**. Contains docs for four products: DevPortal, Admin-UI, Platform, and VKDR-CLI.

## Branching and Deployment

- **`develop`** (default branch) ŌĆö force-pushed to `docs-next` repo ŌåÆ deploys to staging (`https://docs-next.platform.vee.codes/`)
- **`main`** ŌĆö deploys to production (`https://docs.platform.vee.codes/`)

Feature branches should be created off `develop`. PRs target `develop`. Merge `develop` ŌåÆ `main` for production releases.

## Commands

```bash
yarn start      # Local dev server with hot reload
yarn build      # Production build to ./build
yarn serve      # Serve the production build locally
yarn clear      # Clear Docusaurus cache (useful when builds behave unexpectedly)
```

Requires Node.js >= 20. Uses Yarn 4 via Corepack (`corepack enable` to activate).

## Architecture

### Multi-instance docs

The site uses Docusaurus multi-instance docs (not a single `docs/` folder). Each product has its own content directory at the repo root, configured as a separate `plugin-content-docs` instance in `docusaurus.config.js`:

| Directory    | Route base   | Plugin ID   |
|-------------|-------------|-------------|
| `devportal/` | `/devportal` | default     |
| `platform/`  | `/platform`  | `platform`  |
| `admin-ui/`  | `/admin-ui`  | `admin-ui`  |
| `vkdr/`      | `/vkdr`      | `vkdr`      |

All four instances share a single `sidebars.js` (autogenerated from directory structure).

### DevPortal versioning (V1 / V2)

The `devportal` instance is versioned (the others are not). Current posture while the `devportal-platform` rollout is in progress:

- **V1 is the default, served at the root** `/devportal/` ŌĆö the split-image / profiles release that most production and SaaS installs run today. Source: `versioned_docs/version-v1/`.
- **V2 is an opt-in preview** at `/devportal/v2/` ŌĆö the unified `veecode/devportal:2.0.0` image / presets. Source: the `devportal/` current tree. Labeled `v2 (preview)` with an `unreleased` banner.
- This is set in `docusaurus.config.js` (`lastVersion: "v1"`, the `versions` block, paths swapped). When the rollout matures, flip `lastVersion` to `current`, swap the `path` values back, set V1's banner to `unmaintained`, re-point the cross-instance `/devportal/v2/...` links to root, and restore the oldŌåÆnew redirects.

**Freeze rule:** `versioned_docs/version-v1/` is **correctness-only** ŌĆö no new content, only factual fixes. All new work lands in the `devportal/` (V2) current tree. The only deliberately shared transition surface is `devportal/migrating-from-v1.md` and `devportal/which-version.md`. This avoids double-maintenance drift between the two versions.

**Links:** inside the V2 tree, use **relative** `.md` links or `/devportal/v2/...` so they resolve to V2 pages (a bare `/devportal/...` now resolves to V1 at root). Cross-instance links (`platform/`, `admin-ui/`) into DevPortal use `/devportal/v2/...`.

### Custom pages and components

- `src/pages/` ŌĆö Custom landing page (`index.js`)
- `src/components/` ŌĆö Shared React components (header, footer, button)
- `src/css/custom.css` ŌĆö Global theme overrides

### Notable plugins/features

- **Mermaid** diagrams enabled via `@docusaurus/theme-mermaid`
- **Image zoom** via `docusaurus-plugin-image-zoom` (add class `zoomable` to images)
- **Client redirects** configured in `docusaurus.config.js` for legacy URL compatibility

## Content Conventions

- Each doc section has an `intro.md` as its landing page
- Sidebar ordering is autogenerated from filesystem structure (use numeric prefixes or `_category_.json` files to control order)
- Documentation is English-only (`i18n` configured for `en`)
